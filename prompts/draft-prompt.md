# 下書き生成プロンプト

以下の骨子をもとに、記事の下書きを生成してください。

## 入力情報
- 骨子: {{outline}}
- ペルソナ: persona.mdを参照

## ルール（厳守）
1. 文体：です・ます調、一人称「僕」、断定型
2. 各h2セクションはPREP法で構成（Point→Reason→Example→Point）
3. Exampleパートで実体験が必要な箇所は `[EXPERIENCE: 説明文]` のプレースホルダーを残す。絶対に実体験を捏造しない
4. h2直下の最初の1文は結論（Point）を述べる
5. CTAボタンは `[CTA_BUTTON:エージェントキー]` の形式で挿入する
6. 1セクション300〜500文字程度
7. 専門用語は初出時にカッコ書きで簡潔に説明する
8. **太字（`**...**`、HTML描画後は `<strong>`）の戦略的使用（AEO/GEO対策）**
   - 目的：AIが記事を要約・抜粋する際、太字部分が優先的に拾われる傾向があるため、情報の重要度に応じて意図的に強弱をつける。
   - 太字にする対象は、次の「本当に重要な情報」に**限定**する。
     - (a) **各セクションの結論**：PREP法のP、すなわちh2直下の最初の1文のうち、**結論の核心となる語句だけ**（文全体を囲まない）。
     - (b) **数値・固有名詞**：応募社数・年収・手数料率などの数字、サービス名、統計データの数値。
   - **多用しない。1セクションにつき1〜2箇所まで**。
   - **見出し的な強調（ラベル付け）に太字を使わない**（例：`**メリット**` のような小見出し用途は不可。箇条書きのラベルも同様）。AIが抜き出すべき核となる事実・結論のみを対象にする。
   - `==...==`（蛍光マーカー＝ `<mark>`）とは役割が異なる。AI抽出を狙う核心には `**`（`<strong>`）を優先し、`==` の多重掛けは避ける。
   - **日本語の記号隣接に注意**：CommonMarkの仕様上、`**` で囲む語句が**記号で始まる/終わる**と太字が効かず `**` がそのまま表示される（例：`**約75〜80%**` の末尾 `%`、`**「3年以上4年未満」**` の前後 `「」`）。この場合は次のいずれかにする。
     - 記号を `**` の外に出す（例：`「**3年以上4年未満**」`）。
     - もしくは生HTMLタグ `<strong>約75〜80%</strong>` を使う（本文Markdownは `sanitize:false` で描画されるため `<strong>` はそのまま有効）。
     - 数字・かな・漢字・英字で始終わる語句（例：`**120〜140万円**`）はそのまま `**` でよい。
   - **FAQ回答文には太字（`**` / `<strong>`）を入れない**：現行の `FAQSection` コンポーネントは回答をプレーンテキストで描画し、同じ文字列を構造化データ（FAQPage）にも渡すため、マークアップを書くと画面に記号が露出し構造化データも汚れる。FAQ回答冒頭の一文を強調表示したい場合は、サイト構築エージェント側で「強調をレンダリングしつつJSON-LDにはプレーンテキストを渡す」改修が必要（→ 下記の申し送り事項）。

## サイト構築エージェントへの申し送り事項
- **FAQ回答の強調表示対応**：AEO/GEO観点で「FAQ回答の最初の一文」を `<strong>` で強調したい。ただし表示用（`FAQSection.tsx`）は強調をHTMLとして描画し、構造化データ用（`JsonLd.tsx` の FAQPage `text`）は**タグを除去したプレーンテキスト**を渡すこと。両立できる実装が入るまで、コンテンツ側はFAQ回答に強調記法を書かない。
